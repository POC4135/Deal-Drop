# Branch: fix/phase-c-db-migration

## Summary of Changes from Main

### 1. Database — Full Schema Migration
The Supabase instance was on an old pre-Phase C schema (UUID IDs, different column names, missing tables). All Phase C API endpoints were failing with schema errors (e.g. `column vl.address does not exist`).

**What we did:**
- Dropped all existing tables and enum types in Supabase
- Applied all 3 Phase C migrations via psql:
  - `0001_platform_foundation.sql` — full schema rebuild
  - `0002_production_alignment.sql` — production alignment changes
  - `0003_google_place_metadata.sql` — Google Place metadata columns
- Seeded 19 venues + 77 listings from `ATLDEALS-5-4.xlsx` using a new import script

### 2. New Excel Import Script (`services/api/src/db/import.ts`)
Rewrote the import script from `feat/backend` to work with the Phase C schema.

**Key differences from old script:**
- Uses `pg` client directly (not Drizzle ORM) to match Phase C conventions
- Maps old Excel columns (`address_line_1`, `city`, `state`) → single `address` field in `venue_locations`
- Maps old `category` enum → `category_label` text field
- Generates `varchar(64)` IDs (`ven_*`, `loc_*`) instead of UUIDs
- Uses `ulid()` for listing IDs to avoid slug-truncation collisions
- Builds `schedule_summary` from days + times
- Handles Supabase session pooler connection (explicit params, SSL)

**Usage:**
```bash
cd services/api && pnpm db:import /path/to/file.xlsx
```

### 3. Seed Script Fix (`services/api/src/db/scripts/seed.ts`)
The `pnpm db:seed` script was failing against Supabase with `role "postgres" does not exist`.

**Root cause:** Node.js `pg` client incorrectly parses the Supabase pooler username `postgres.projectref` from a connection string URL, stripping the project ref.

**Fix:** Parse the `DATABASE_URL` manually with a regex and pass explicit connection params (`host`, `user`, `password`, `port`, `database`) to the `pg` Client instead of a connection string. Also added `dotenv/config` import since the script wasn't loading `.env`.

### 4. iOS Deployment Target (`apps/mobile_flutter/ios/`)
`firebase_core` 12.12.0 requires iOS 15.0 minimum but the project was targeting 13.0.

**What we changed:**
- `ios/Podfile` — set `platform :ios, '15.0'`
- `ios/Runner.xcodeproj/project.pbxproj` — bumped `IPHONEOS_DEPLOYMENT_TARGET` from 13.0 to 15.0 in all 3 build configurations

## Still Needed from Colleague

### 5. `GoogleService-Info.plist` (iOS blocker)
Firebase requires this file to initialize on iOS. Without it the app crashes on launch.

- **Where to get it:** Firebase console → Project Settings → iOS app → Download `GoogleService-Info.plist`
- **Where to place it:** `apps/mobile_flutter/ios/Runner/GoogleService-Info.plist`

### 6. Google Maps API key in `Info.plist` (iOS blocker)
`--dart-define=GOOGLE_MAPS_API_KEY` works on web but not iOS. The native Google Maps SDK requires the key embedded in the iOS config.

- **Where to add it:** `apps/mobile_flutter/ios/Runner/Info.plist`
- **Key:** `GMSApiKey`
- **Value:** the Google Maps API key

### 7. Render build command fix (production blocker)
The Render dashboard build command runs `pnpm install` from the wrong directory, so workspace packages (`@dealdrop/config`, `@dealdrop/shared-types`) don't resolve.

- **Fix in Render dashboard → Settings → Build Command:**
  ```
  pnpm install && cd services/api && pnpm build
  ```
  *(must run `pnpm install` from repo root first)*
